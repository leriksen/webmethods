Required commands and useful automation tips for webMethods build and deploy using deployer.sh and projectautomator.sh on Linux, including typical scripting practices, environment setup, and verification commands.

🧩 Overview 

There are two main command-line utilities for automating deployments in webMethods Deployer: 

Tool 

Purpose 

Path 

projectautomator.sh 

Build Deployer project (archive) — creates the deployable build from source repository or IS packages. 

$WM_HOME/Deployer/bin/projectautomator.sh 

deployer.sh 

Deploy project archive to target environments (e.g., Dev → QA → Prod). 

$WM_HOME/Deployer/bin/deployer.sh

Before running either script: 

cd /webMethods/Deployer/bin

export JAVA_HOME=$WM_HOME/jvm/jvm 

export PATH=$JAVA_HOME/bin:$PATH

java -version

4️⃣ Common Automation Sequence (End-to-End Script) 

Here’s how a full CI/CD job typically chains commands: 

#!/bin/bash 

# Example: deployer_build_deploy.sh 

 

export WM_HOME=/webMethods 

export DEPLOYER_HOME=$WM_HOME/Deployer 

export PATH=$JAVA_HOME/bin:$PATH 

cd $DEPLOYER_HOME/bin 

 

echo "=== Building Deployer Archive ===" 

./projectautomator.sh -input /opt/wmprojects/config/buildProject.xml -log /opt/wmprojects/logs/build.log 

if grep -q "BUILD SUCCESSFUL" /opt/wmprojects/logs/build.log; then 

    echo "Build successful" 

else 

    echo "Build failed"; exit 1

echo "=== Deploying Archive to QA ===" 

./deployer.sh -input /opt/wmprojects/config/deployProject.xml -log /opt/wmprojects/logs/deploy.log 

if grep -q "Deployment completed successfully" /opt/wmprojects/logs/deploy.log; then 

    echo "Deployment successful" 

else 

    echo "Deployment failed"; exit 1 

fi

Build.properties:  

 

webMethods Deployer automation For repository-based deployments (i.e., building directly from VCS/Asset repository instead of from a live Integration Server). 

 

The build.properties file is a key configuration file used by webMethods Deployer when you create builds (archives) from a repository, such as: 

Asset Build Environment (ABE) outputs, 

version control repositories (Git, SVN, etc.), 

or exported packages/assets stored locally. 

 

It provides metadata and paths to tell Deployer where to find build artifacts, configuration files, and repository settings. 

 

àIn short — build.properties bridges the build artifacts (from repository) and the Deployer build script (projectautomator.sh), so the build can be created automatically without connecting to a live IS source. 

 

Typical Location 

/webMethods/Deployer/projects/<ProjectName>/build.properties 

 

Or 

 

/opt/wmprojects/deployer/build.properties 

 

You can also pass it explicitly to projectautomator.sh using: 

./projectautomator.sh -buildProperties /opt/wmprojects/config/build.properties 

 

📄 Typical Contents of build.properties 

Here’s a realistic, commented example of what build.properties contains for a repository-based deployment: 

 

############################################################################### 

# webMethods Deployer - build.properties 

# Used for repository-based builds (no live Integration Server connection) 

############################################################################### 

 

# Repository Configuration 

repo.type=local               # or vcs (git/svn) or assetRepo 

repo.path=/opt/wmprojects/repository/export/ISPackages 

# repo.url=https://git.company.com/scm/webmethods.git 

# repo.branch=release/1.0.0 

# repo.username=gituser 

# repo.password=encrypted:<encrypted_pwd> 

 

# Project Configuration 

project.name=CommonFramework_Deploy_Project 

project.folder=/opt/wmprojects/deployer 

 

# Archive Metadata 

archive.name=CommonFramework_Archive_1.0 

archive.description=Deployment archive for Archive Management IS Packages 

archive.output.dir=/opt/wmprojects/deployer/archives 

 

# Deployment Set Details 

deployments.set.name=CommonFramework_Set 

deployments.set.description=Contains ArchiveProcessing, ArchiveUtils, and Config packages 

 

# Assets to Include (paths are relative to repo.path) 

packages=ArchiveProcessing,ArchiveUtils,ArchiveConfig 

config.files=config/Archivedb.properties,config/logging.properties 

 

# Target Alias (for deployer mapping later) 

target.alias=QA_IS 

 

# Build Output Control 

build.log.dir=/opt/wmprojects/logs 

build.timestamp=true 

overwrite.archive=true 

 

# Optional — Versioning / tagging info 

build.version=1.0.0 

build.label=Release_1.0 

 

How It Helps — Purpose of Each Section 

Section 

Purpose 

Repository Config 

Tells Deployer where to pull the buildable components (e.g., IS packages, configurations, schemas). No need to connect to source IS. 

Project Config 

Defines the Deployer project name and folder to place the build metadata. 

Archive Metadata 

Defines how the output archive (the .zip) will be named and stored. 

Deployment Set Details 

Defines what logical set(s) of assets to include in this build (similar to a Deployment Set in the Deployer UI). 

Assets to Include 

Specifies the actual packages, configurations, and other deployable assets to package. 

Target Alias 

Used for pre-mapping the deployment to a target alias (optional, for convenience). 

Build Output Control 

Defines log/output directories, timestamping, and overwrite behaviors. 

Versioning 

Optional — helps maintain traceability and CI/CD automation. 

 

How It Works During Build 

When you run: 

./projectautomator.sh -buildProperties  /opt/wmprojects/config/build.properties -log build.log 

 

Internally, Deployer does: 

Reads the build.properties file to locate repository artifacts. 

Loads the project definition (if not existing, creates one). 

Includes the listed IS packages/config files into a deployment set. 

Builds the Deployer Archive (.zip) under: 

 

/IntegrationServer/replicate/deployer/archive/ 

 

or the directory defined in: 

archive.output.dir 

 

Logs details to /opt/wmprojects/logs/build.log.

Integration with ABE (Asset Build Environment) 

If you use Asset Build Environment (ABE) to generate artifacts from a CI/CD pipeline, ABE produces output in: 

 

/opt/wmprojects/repository/export/ 

 

Then you can point your build.properties like: 

repo.type=local 

repo.path=/opt/wmprojects/repository/export/ 

packages=MyAPIService,ArchiveProcessing 

 

This allows Deployer to build directly from ABE outputs, completing a repository-based deployment cycle without needing any runtime servers.

1️⃣ deployProject.xml — Automated Deployment Template 

This XML file is consumed by deployer.sh and instructs webMethods Deployer how to deploy the built archive to the target Integration Server(s). 

📄 Path: /opt/wmprojects/config/deployProject.xml 

<?xml version="1.0" encoding="UTF-8"?> 

<Deployer> 

  <DeployProject name="${{ variables.projectName }}" folder="${{ variables.projectFolder }}"> 

     

    <!-- Archive previously created by projectautomator.sh --> 

    <Archive name="${{ variables.archivePrefix }}_${{ variables.buildVersion }}_${{ variables.buildId }}"/> 

 

    <!-- Target mapping --> 

    <MapTargets> 

      <Map alias="${{ variables.targetAlias }}" 

           host="${{ variables.targetHost }}" 

           port="${{ variables.targetPort }}" 

           user="${{ variables.targetUser }}" 

           password="${{ variables.targetPassword }}" /> 

    </MapTargets> 

 

    <!-- Deployment Set(s) to deploy --> 

    <DeployDeploymentSet name="${{ variables.deploymentSetName }}" /> 

 

    <!-- Optional: pre/post actions --> 

    <Simulate>false</Simulate> <!-- set true for dry run --> 

    <RestartAfterDeploy>false</RestartAfterDeploy> 

    <ContinueOnError>false</ContinueOnError> 

  </DeployProject> 

</Deployer>

#!/bin/bash 

host=$1; port=$2; user=$3; pass=$4

    echo "Target IS ping failed! Check deployment." 

    exit 1
